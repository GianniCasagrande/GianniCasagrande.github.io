Search & Data Filtering, Mobile

Research • Concept Creation • Visual Design • Prototyping • User Testing

About

The Axonify platform gives managers and administrators great insight into the knowledge and health of your workforce training. Managers and admins can view and report on data based on up to 5 different criterias. Lots of options and insights to say the least.

Problem

While working on a page performance issues, server developers discovered that a single user had created 800 individual custom filters. This was hurting page load. Each filter was setup to view 1 single team. Why? Well a huge product gap was found, there was no easy way to search or filter for a singular team without creating a “custom filter” for that view. Not good. Custom filter creation is very intensive in the current iteration. All the user wants to do is view a team or two.

All these custom filters only have one team...that's alot of filters

Constraints

Could not abandon current functionality. Users still needed to create custom filters based on all five data types. We also did not want to break functionality for current users. We just needed to solve our main use case, quickly filter on one or multiple teams.

Scope Creep

Currently, all the "zones" were using different filter components to search and filter data. It was decided that we were going to try to create a single component that could be re-used globally over all “zones”. Nail down mobile fist, then we would port it over to a desktop experience as well. We were opening up the guts anyway!

High-Level Asks

  • Ability to find a singular item easily.
  • Filter one to many teams or items quickly, possibly in an ad-hoc way
  • Mobile friendly first, support for active on-the-go managers
  • Try to create a standalone component that can be used in other “appzones”
Early Sketches

Research

  • Worked with development to visit user’s production environment to see how admins were using the current filtering.
  • With the aid of my PM, touched base with customers to understand why they felt the need to create all these filters.
Low Fed Screens

Solution

After leading some design meetings, it was decided to go with a search first approach. Being able to find a single team quickly was discovered to be pretty much 80% of the use cases. If the user needed to filter on more data, the idea was they could go into the advanced filter mode where they can view, edit or create new filters based on any criteria the platform offered.

After some testing and iterations, it was found that it would be great to filter quickly on any of the criteria easily. I beleive that this would prevent the creation of many custom filters, and if by chance the user found the data on a specific view helpful, the ability to easily save it as a custom filter would still exisit.

Key Experiences

  • Easily find a single item under any criteria in under 3 clicks
  • Filter data quickly on any criteria, without having to create a filter
  • Quickly save any ad-hoc state into a filter you can re-use
  • Selected items and filters persist over page loads
  • Flexible UI design to support 19+ languages
